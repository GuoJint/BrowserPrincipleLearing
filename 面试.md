 项目两点/挑战:
 1. 有个需要时需要与企微进行交互方便业务去做一些快捷操作不用去打开后台，写一个H5页面放在企微侧边栏。独立负责与企微的前端交互，包括企业微信配置及与后端服务的各种签名验证，以及授权问题。当时也是踩了很多坑，比如获取授权后的code需要通过code调用后端接口获取用户信息。但是code只能消费一次，提前消费掉code导致没有正常获取到用户信息。
 2. 需要在一个输入框中插入变量并且预览时需要将变量替换为实际值。当然整个功能还需要拖拽新增等等。可编辑div+contenteditable，最难的部分主要还是对光标的处理，插入变量后需要记录插入前的位置，然后根据插入内容长度，更新光标位置。还有长度限制，如果超出长度插入变量截断，复制内容截断等等的处理。
 即使虚拟化的表格是高效的，但是当数据负载过大时，网络和内存容量也会成为您应用程序的瓶颈。 因此请牢记，虚拟化表格永远不是最完美的解决方案，请考虑数据分页、过滤器等优化方案。

## 首屏渲染优化
1. 文件体积过大情况：使用webpack优化打包体积(tree shaking,代码分割，cssnano),压缩图片
2. 资源加载阻塞页面渲染： 
  - 预加载（preload）：首屏关键资源预加载用 <link rel="preload"> 优先加载首屏必需的资源（如核心 JS、首屏图片）、
  - 添加 defer/async（defer 不阻塞渲染且按顺序执行，async 异步执行）、
  - 预连接 / 预解析(preconnect, dns-prefetch)：提前建立与第三方域名的连接，减少 DNS 解析 / 握手时间 <link rel="preconnect" href="https://cdn.example.com">、<link rel="dns-prefetch" href="https://cdn.example.com">，
  - 使用 <img loading="lazy"> 延迟加载图片：图片出现在可视区域再加载、
3. 网络延迟、服务端响应慢会直接影响首屏资源加载速度 
  - HTTP/2/HTTP/3：启用 HTTP/2（多路复用）或 HTTP/3（QUIC 协议），减少 TCP 握手和资源阻塞。
  - CDN 加速：将静态资源（JS/CSS/ 图片 / 字体）部署到 CDN，利用就近节点加速下载。
  - 缓存策略：
        强缓存：设置Cache-Control: max-age=31536000（永久缓存）、Expires，缓存静态资源；
        协商缓存：设置ETag/Last-Modified，资源未修改时返回 304，避免重复下载。
  - SSR/SSG/ISR（框架层面）：
        SSR（服务端渲染）：React Next.js/Vue Nuxt.js 将首屏 HTML 在服务端渲染完成，返回给浏览器直接渲染，避免客户端渲染的空白期；
        SSG（静态站点生成）：构建时预渲染所有页面，生成静态 HTML，加载速度最快；
        ISR（增量静态再生）：结合 SSG 和 SSR，既保留静态页面的速度，又支持动态内容更新。
4. 即使首屏加载未完成，通过优化用户感知，降低卡顿 / 空白的负面影响：骨架屏（Skeleton Screen）：首屏加载时显示页面骨架（占位符），替代空白页

## 总结
首屏渲染优化的核心要点可总结为 3 点：
    减体积：压缩 / 分割 JS/CSS、优化图片 / 字体，减少资源下载量；
    优顺序：优先加载首屏关键资源，非关键资源异步 / 延后加载，避免阻塞；
    提感知：通过骨架屏、加载态减少用户对延迟的感知，同时结合 CDN/SSR/ 缓存提升网络 / 服务端效率。
实际优化时，建议先通过 Lighthouse 定位核心瓶颈（如 “大体积未压缩图片”“JS 阻塞渲染”），再针对性落地上述措施，避免无目的的全量优化。
